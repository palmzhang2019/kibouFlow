# GEO 后台阶段 3 实施计划：运营化、权限与交接能力

> 目标：从“能配”升级到“能长期运营 + 能交接”。

---

## 1. 阶段定位

| 项目 | 值 |
|---|---|
| 阶段编号 | Backend Phase 3 |
| 预估工时 | 4-7 人日 |
| 优先级 | P1 |
| 主要产出 | 权限体系、审批流、质量看板、交接包 |

一句话目标：让 GEO 后台成为组织资产，而不是个人经验。

---

## 2. 范围

1. 角色权限：管理员 / 编辑 / 审核人。
2. 配置发布流程：草稿 -> 待审核 -> 已发布。
3. GEO 健康看板：缺失项、错误项、覆盖率。
4. 交接导出：配置快照、变更记录、责任人信息。

---

## 3. 看板指标建议

- metadata 完整率（标题、描述、canonical）。
- JSON-LD 覆盖率（Article/FAQ/HowTo/Breadcrumb/WebSite）。
- 抽取失败页数量（FAQ < 阈值、HowTo < 阈值）。
- noindex 页面数量与变化趋势。
- 最近 7/30 天配置变更频次。

---

## 4. 任务清单

| 编号 | 任务 | 验收 |
|---|---|---|
| B3-1 | RBAC 权限控制 | 不同角色看到不同操作按钮 |
| B3-2 | 审批流（草稿/发布） | 未发布配置不影响线上 |
| B3-3 | GEO 健康看板页面 | 可按 locale、目录筛选 |
| B3-4 | 交接导出（JSON/CSV） | 一键导出站点与页面配置 |
| B3-5 | 操作审计日志 | 支持按人/时间检索 |

---

## 5. 风险与控制

- 风险：审批流程过重，降低更新效率。
- 控制：允许紧急发布通道，但必须记录审计日志。
- 风险：权限配置复杂导致维护困难。
- 控制：先固定 3 角色，不做自定义角色系统。

---

## 6. 退出标准

- [ ] 非开发角色可独立完成配置新增、提审、发布。
- [ ] 发布前后可追溯变更记录。
- [ ] GEO 看板可识别“高风险页面”并给出列表。
- [ ] 交接时可导出完整配置与近 90 天变更。
